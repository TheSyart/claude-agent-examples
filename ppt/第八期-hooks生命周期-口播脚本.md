# 第八期 Hooks 生命周期口播脚本

来源: `ppt/第八期-hooks生命周期.html`

建议时长: 8 到 10 分钟

对应代码: `build-agent-example/code/step12_hooks.py`

风格: 面向小白，先讲生活比喻，再给技术名字。少代码，重点讲清楚 Agent 为什么需要 Hooks。

## S0 封面: Hooks 是 Agent 流程里的检查点

这一期讲 Hooks。

Hooks 这个词，中文直译是"钩子"。

但我们先别想钓鱼钩。放到 Agent 里，它更像是流程上的检查挂点。

什么意思？

Agent 本来有一条主路。

用户输入进来，模型思考一下，可能要调用工具，工具返回结果，最后 Agent 给出回答。

如果只看这条主路，很简单。

但真实用起来，你会发现一堆额外问题。

模型想写文件，能不能写？

模型想执行命令，危险不危险？

工具输出太长，要不要截断？

这一轮花了多久，要不要记日志？

这些事情都很重要，但它们又不是 Agent 主流程本身。

所以我们需要在关键位置挂一些检查点。

这些检查点，就是 Hooks。

这期我们不先讲复杂源码。先用一个生活里的场景，把它讲明白。

## S1 第二页: 从"钩子"这个意思切入

先看"钩子"这个词。

生活里，钩子最大的作用不是自己干活。

它是挂东西的。

墙上有一个钩子，你可以挂钥匙，挂包，挂帽子。

墙还是那面墙，走廊还是那条走廊，但你多了一个可以挂规则、挂动作的位置。

Agent 里的 Hook 也是这个意思。

它不是另一个 Agent。

它也不是让模型变聪明的魔法。

它就是在 Agent 主流程上留出来的挂点。

用户输入前，可以挂一个检查。

工具执行前，可以挂一个检查。

工具执行后，可以挂一个记录。

准备结束时，可以挂一个收尾判断。

这就像一条地铁进站线。

人往前走，流程往前跑。

每到一个关键位置，都可以挂一张规则卡。

这张卡可以说：放行。

也可以说：停一下，这个动作危险。

还可以说：路径换一下，别写到生产目录，写到沙箱里。

这里先记一句话。

Hook 不是主流程。

Hook 是挂在主流程旁边的检查点。

## S2 第三页: 引入 Agent Hooks

那为什么 Agent 特别需要 Hooks？

因为 Agent 不只是聊天。

普通聊天模型，只是生成文字。

Agent 会做动作。

它可能读文件、写文件、执行命令、调用接口、派子代理。

动作一多，风险就来了。

你不能只相信模型说"我会小心"。

模型可以提出动作，但真正执行动作的是外面的运行环境。

也就是 Runner。

所以安全检查要放在 Runner 这里。

换成地铁安检的比喻，就是乘客想进站，但能不能进，要过闸机和安检。

模型想调用工具，但能不能调用，也要过检查点。

Agent Hooks 就是在这些站点上挂规则。

输入站点，可以补充上下文。

模型站点，可以记录耗时。

工具站点，可以检查参数。

结束站点，可以判断任务是不是该收尾。

这些规则不一定都很复杂。

有些只是记录："谁在什么时候调用了写文件工具"。

有些是门禁："不要写 `.env`"。

有些是改写："你要写 `demo_production`，我帮你改到 `sandbox/demo_production`"。

有些是人工确认："你要 `git push`，先问一下人"。

这就是 Agent Hooks 的价值。

它让 Agent 有了流程边界。

## S3 搭建骨架: Hook / Registry / Runner

现在开始讲怎么搭。

我们只需要三个东西。

第一个叫 Hook。

你可以把它理解成一张规则卡。

这张卡写着：我关心哪个位置，我要检查什么，检查完给什么结果。

比如一张规则卡写着：

只要有人要写 `.env`，就拒绝。

另一张规则卡写着：

只要工具执行完，就把结果长度记录一下。

第二个叫 Registry。

这个词听起来有点硬。

你可以把它理解成挂规则卡的架子。

所有 Hook 都挂到这个架子上。

流程走到某个站点，Registry 就按顺序把规则卡拿出来问一遍。

第一张卡看一下。

第二张卡看一下。

如果某张卡说"停"，后面的卡就不用再问。

第三个叫 Runner。

Runner 就是 Agent 主流程。

它负责接收用户输入、调用模型、执行工具、回填结果。

Runner 每走到一个关键位置，就喊 Registry：

这里有人要过站了，你们这些规则卡看一下。

所以这三个词不用背复杂定义。

Hook 是规则卡。

Registry 是挂卡架。

Runner 是过站的主流程。

代码里的 `step12_hooks.py` 就是按这个结构写的。

## S4 挂点放在哪里

接下来讲挂点。

Hook 不能乱挂。

它必须挂在流程边界上。

教学代码里有几个常见位置。

第一个是 `before_turn`。

一轮对话开始前触发。

这时候还没调用模型，所以适合做上下文注入、限流、系统提示词补充。

第二个是 `after_turn`。

模型这一轮结束后触发。

这时候适合记录耗时、记录模型输出。

第三个是 `before_tool_call`。

工具执行前触发。

这是最关键的位置。

因为工具还没真的动手。

要拦危险命令，要保护敏感文件，要让用户确认，都应该放这里。

第四个是 `after_tool_call`。

工具执行后触发。

这时候动作已经发生了，所以它更适合审计、统计、格式化输出。

比如输出太长，可以截断一下再交给模型。

第五个是 `on_stop`。

Agent 准备结束时触发。

它可以做一个最后检查。

比如发现任务还没完全收尾，就提醒 Agent 再补一轮。

用地铁线来说：

进站前可以拦。

进站后只能记录。

这个区别很重要。

## S5 一张 Hook 卡怎么工作

一张 Hook 规则卡，内部也不用想复杂。

准确说，是三问一结果。

前三个是问题，最后一个是结果。

第一个问题：谁路过了？

这就是 Event。

比如现在是工具执行前，还是工具执行后。

第二个问题：这事归我管吗？

这就是 Matcher。

比如我是危险命令检查，那我只关心 `run_command`。

如果路过的是 `read_file`，我就不管。

第三个问题：我要怎么处理？

这就是 Handler。

Handler 会看上下文。

比如工具名是什么，参数是什么，要写哪个路径，要执行哪条命令。

处理完以后，它才给一个结果。

这个结果叫 Decision。

Decision 也很好理解。

`allow`，放行。

`deny`，拒绝。

`ask`，问人。

`block`，更硬一点的阻断。

有时还会带一个改写后的输入。

比如模型说要写 `demo_production/report.txt`。

Hook 可以把它改成 `sandbox/demo_production/report.txt`，再放行。

这就是 `updated_input`。

所以完整结构不要说成四个问题。

它是三问一结果：

谁来了，归不归我管，我怎么处理。

最后返回什么决定。

如果你理解了这句话，Agent Hooks 的骨架就通了。

## S6 常见结果演示

这一页讲几个真实演示。

第一个是审计。

模型调用了写文件工具。

工具执行完之后，Hook 记录一行日志。

这个日志可以写工具名、输入参数、执行时间、结果状态。

这类 Hook 不改变流程，只是让事情看得见。

第二个是沙箱改写。

模型想写 `demo_production/report.txt`。

这听起来像生产目录。

我们不直接拒绝，也不直接写。

Hook 把路径改成 `sandbox/demo_production/report.txt`。

动作还是完成了，但风险小很多。

第三个是破坏性命令拒绝。

如果命令命中了破坏性模式，工具还没执行，Hook 就直接拒绝。

这里不要等 After Hook。

等执行完再记录，已经晚了。

第四个是人工确认。

比如提交、发布、部署这类会影响外部状态的动作。

这类动作不一定永远危险，但影响比较大。

所以 Hook 可以问终端：确定要执行吗？

用户输入 `y` 才放行。

没人确认，就默认拒绝。

第五个是输出截断。

有些工具输出很长。

如果原样塞回模型，上下文会被撑爆。

After Hook 可以把输出裁短，保留前面重要部分，再告诉模型："后面省略了"。

这几个例子覆盖了大部分教学场景。

看见、改写、拒绝、询问、整理输出。

## S7 安全边界

这里要讲清楚一个坑。

不是所有 Hook 都能做安全边界。

After Hook 听起来也能检查，但它发生在工具执行之后。

如果命令已经跑了，文件已经删了，你再说"这个危险"，只能写事故报告。

所以安全规则要尽量放在 Before。

工具执行前拦住，才是真的拦住。

还有一个坑。

不要把 LLM 当硬边界。

你可以让模型帮忙判断语义。

比如回答有没有跑题，任务有没有完成。

但涉及删文件、发请求、推代码、付款这类动作，最好还是用明确规则和人工确认。

白话一点说：

LLM 可以当顾问。

门锁不能只靠顾问。

Agent Hooks 的价值，就是把这些硬规则放到运行环境里。

模型可以想，Runner 负责执行，Hook 负责把关。

这三件事分开，系统才不会乱。

## S8 收尾复盘

最后复盘一下。

Hooks 直译是钩子。

放到 Agent 里，它不是钓鱼钩。

它是流程上的检查挂点。

Agent 主流程继续往前跑。

Hook 挂在关键站点旁边。

它可以记录，可以改写，可以拒绝，也可以问人。

实现上记住三个角色。

Hook 是规则卡。

Registry 是挂卡架。

Runner 是主流程。

一张规则卡内部，记住三问一结果。

三问是：谁路过，归不归我管，我怎么处理。

结果是：返回什么决定。

这就是 Event、Matcher、Handler、Decision。

如果你是从零做 Agent，我建议顺序很简单。

先做日志。

让所有动作看得见。

再做参数改写。

把明显危险的路径导到沙箱。

然后做门禁。

敏感文件、危险命令，直接拒绝。

再往后，才是人工审批和语义验收。

不要一上来就做很复杂的 Agent-Based Hook。

先把硬规则做稳。

这期对应代码在 `build-agent-example/code/step12_hooks.py`。

你可以从普通写文件开始试，再试沙箱改写、危险命令拒绝、人工确认和输出截断。

跑完这些例子，Hooks 就不是一个抽象词了。

它就是 Agent 流程里的检查点。
