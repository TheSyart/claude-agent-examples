DEFINE 02 · ROLE & POSITION
子代理是工具箱里
"上下文隔离"这个原语
在工程代码里，子代理以 dispatch_subagent 工具出现。它不是平级协作者，而是主 Agent 调用的一个特殊 tool：内部临时跑一个完整 AgentRunner。
是
主 Agent 的一个工具
主帧里以 tool_use 调用, 主 history 只多一条 tool_result
是
一段临时 AgentRunner
独立 history、独立 system、独立白名单, 跑完即焚
非
不是平级协作者
主→子单向调度, 子不能再派子, 也不能改主 todo / 主 history
关注点主 Agent子代理
身份调度者 · 决策者执行者 · 搜集者
对谁负责对用户的下一句回答对一段 task 描述
生命周期跨多轮对话一次调用即结束
权限边界全工具 + dispatch + todo子集白名单, 无 dispatch · 无 todo
history持久 · 三层记忆 · 会被压缩临时, 不入主 history
失败影响面主对话中断仅一条 tool_result 回传 Error
主线增量—1 条 tool_result (一段中文汇报)
CMP
抓网页、读文件、跑命令都可以在子 history 里完成；主 history 只得到最终摘要。这就是上下文隔离的收益。

FLOW 06 · END TO END
从用户一句话
到子代理回禀
理解子代理最好的方式，是跟着一条调用链走：主模型发 dispatch，工具创建子 runner，子 runner 自己循环用工具，最后结果压缩回主线。

User 给主 Agent 下旨
“派东厂探事去抓两个网页并总结差异”
2
主模型 emits dispatch_subagent
agent_type + task + purpose
3
Dispatch 工具启动子 runner
临时 history = [{role:user, content:task}]
4
子 runner 内部 tool_use 循环
web_fetch / grep / read_file 输出留在子上下文
5
final summary 回填主 history
主线只多一条 tool_result

// concurrency  ·  并发子代理提高效率
CONCURRENCY 05 · PARALLEL DISPATCH
并发不是命令
是同帧多个 tool_use
工程化版本在 AgentRunner._execute_tool_blocks 里做并发分组：连续多个 concurrency_safe 工具会并行执行，再按原顺序回填。
↑
该并发
差事互不依赖、读多写少、I/O 等待时间长
↓
不该并发
后步依赖前步、共享可写资源、需要顺序确认
TRIGGER
只有模型同一轮真的发出多个 dispatch_subagent 时才会并发。简单任务可能被模型用一条 run_command 直接办完。

DESIGN 08 · SAFETY & TRADEOFFS
子代理能干活
但不能乱管家
工程化版本把权限收紧在代码层，而不是只靠 prompt 祈祷。身份可以活泼，边界必须硬。
No Recursive Dispatch
白名单不含 dispatch
子代理不能继续派子代理, 防止无限嵌套。
No Main Todo
白名单不含 update_todos
计划状态归主 Agent 管, 子代理只交局部结果。
Max Turns
每种身份有轮数上限
探索失败要回报, 不能无限试错。
Model Routing
并发由模型触发
模型没发多个 dispatch, 就不会并发。
