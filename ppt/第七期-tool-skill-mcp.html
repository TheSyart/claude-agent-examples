Tool
能真动手的一台厨具
菜刀切、炒锅煮、秤称、烤箱烤——
给料下去，出结果；
一件事一台厨具，越专一越好。

Skill
把多步骤编成顺序的菜谱
比如红烧肉 SOP：
先炒糖色、下肉煸炒、再加料焖煮。
菜谱不动手，只安排先后顺序。
📱
MCP
手机叫外卖，到了就能用
自家厨房做不了的，叫外卖解决。
外卖到了就是能用的菜，
不用自己再开一个厨房。
// 第 4 页  —  MCP 怎么被调用
Model Context Protocol · 外部工具怎么接进来
外面的工具，怎么接过来调
Anthropic 2024 年底发布的开放协议。以前每个 Agent 接 GitHub、数据库都要自己写对接代码，格式各不同、维护成本高。
MCP 定了统一的"对话语言"：Server 按协议暴露能力，Client（Agent）按协议发现和调用。
Host
装着模型的 Agent 应用
MCP Client
① 启动：schema →
② 调用：tool call →，结果回
Server
外部工具（GitHub、时间…）
stdio / HTTP
// 启动握手
Server 报 schema，Agent 收下翻译
Server 把"我能做哪些工具"的 schema 递过来，Agent 收下、翻成模型认识的工具格式，注册进可用 Tool 列表。
// 真调用
Agent 发 tool call，MCP 转交执行
模型发出 tool call，Host 经 MCP 把请求转给 Server 真执行，结果原路端回模型。
启动报 schema · 转成 Tool · tool call 转交执行
辨
Skill 是"菜谱"，管怎么组合 Tool；MCP 是"手机接单"，管外部 Tool 怎么接进来——一个管做法，一个管连接。
// 第 5 页  —  Skill 怎么被调用
Skill 怎么被加载
用到哪个 Skill
才加载那个
Skill 是描述 Agent 行为的 SOP 文件：告诉 Agent 分哪几步、依赖哪些 Tool、输出什么格式。自己不执行，执行交给 Tool。Agent 不会把所有 Skill 全塞进上下文——分三层按需加载。
名
按需加载这套机制有个正经名字叫渐进式披露（progressive disclosure），图的就是省上下文 token，别一上来就把全部 Skill 都塞给模型。
渐进式披露 · progressive disclosure
System Prompt 里只有这一行
① 菜单行（始终加载）
red-braised-pork — 制作红烧肉的 SOP
几乎不占 token，Agent 靠这一句判断要不要激活这个 Skill。
↓ 决定要用，才读入完整文件
读入完整 SKILL.md
② 完整定义（命中才加载）

辨
谁在调它？Skill 按步骤指挥，一个 Skill 能依次调好多 Tool。分不清 Skill 还是 Tool？问它会不会真执行——会执行就是 Tool，只安排顺序就是 Skill。

// 第 9 页  —  真能跑
动手试试 · MCP
配一行
就接进来
想用一个"报时间"的外部工具？不用写代码——在配置里写一行，启动时它就像外部厨房报了菜单，模型直接能点。
mcp_servers.json
"time": { "command": "python",
       "args": ["…/mcp_server_time.py"] }
跑起来看看
$ python step10_mcp.py
[MCP] 已连接 'time'，多了 2 个工具

你：现在几点？
它：[调用 mcp_time_…]
现在是 18:14:44，皇上。
不用写代码 配一行就行 接进来像自家厨具
一句话收尾
Tool 是能执行的一台厨具，Skill 是知道怎么组合厨具的菜谱，MCP 是手机线上接单；
真正决定先后顺序、何时调用、何时停手的，是 Agent。
