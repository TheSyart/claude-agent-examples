// 机制  ·  Registry 与短路
HookRegistry
按顺序触发
非空即短路
`HookRegistry.emit(event, ctx)` 遍历所有 hook；任一 hook 返回非 None，立即中断并返回该值。这让限流 hook 能直接拦截调用。
顺
顺序执行
按注册顺序调用每个 hook 的同名方法
短
短路返回
返回非 None 时跳过剩余 hook
隔
错误隔离
单个 hook 异常不影响其他 hook 和主循环
RateLimitHook
def before_turn(self, ctx):
  now = time.monotonic()
  self._timestamps = [
    t for t in self._timestamps
    if t > now - self.window_sec
  ]
  if len(self._timestamps) >= self.max_calls:
    return "Rate limit exceeded"
  self._timestamps.append(now)
返回字符串后，`AgentRunner.step()` 直接将该字符串作为本轮结果返回，不再调用 LLM。

// 形态  ·  Before / After / Around
Hook 三种形态
不只是观察
还能修改和拦截
判断框架中的某个机制是否算 Hook，不看名字，看三个问题：能否修改输入？能否阻止操作？能否替换输出？
前
Before Hook
操作前运行：参数校验、权限检查、敏感操作拦截、上下文注入
后
After Hook
操作后运行：日志追踪、格式化输出、审计记录、发送通知
裹
Around / Wrap Hook
包裹操作：短路、重试、缓存、切换模型、结果替换——能力最强

// 辨析  ·  Hook vs Prompt / Tool / Guardrail
概念对比
名字不重要
看具体能力
各框架用词不统一：Claude Code 叫 Hooks，OpenAI 叫 AgentHooks+Guardrails，LangChain 叫 Middleware，CrewAI 叫 Execution Hooks。关键不是名字，而是三个问题。
1
Hook vs Prompt
Prompt 是给模型看的指令（依赖遵循）；Hook 是 Runtime 执行的规则（实际边界）
2
Hook vs Tool
Tool 由模型选择调用；Hook 由系统在事件发生时自动触发
3
Hook vs Guardrail
Guardrail 描述策略约束（是什么）；Hook 描述执行节点（在哪做）
Prompt vs Hook
Prompt："请不要删除生产数据库"
→ 模型可能"忘记"遵循

Hook：检测到 DELETE production → DENY
→ Runtime 直接拒绝，不可绕过
实际框架对照：Claude Code (Hooks)、OpenAI (AgentHooks+Guardrails)、LangChain (Middleware)、Google ADK (Callbacks)、CrewAI (Execution Hooks)、Semantic Kernel (Filters)——解决的是同一类问题，只是术语不同。
// 警示  ·  不是所有 Hook 都是安全边界
反直觉的四个事实
Hook 不等于
安全机制
⚠
After Hook 无法阻止操作
等 after_tool 日志发现危险时，操作已经发生。权限控制必须放 Before，同步执行，fail-closed。
⚠
异步 Hook 不适合授权
后台日志/通知类 Hook Agent 不会等待结果。适合 telemetry，不适合支付审批、数据外发检查。
⚠
LLM Hook 不能当绝对安全边界
可能被提示注入、忽略隐晦风险、相同输入给出不同判断。安全遵循：硬规则 → 权限 → 人工 → LLM，不可反过来。
⚠
Hook 脚本自身是攻击面
Hook 配置应视为可执行代码——检查来源、最小权限、不硬编码密钥。
安全的 Hook 分层
第一层 确定性硬规则
    路径白名单、金额上限、正则匹配
第二层 权限系统
    RBAC、fail-closed、同步执行
第三层 人工审批
    删数据、发布生产、付款
第四层 LLM 语义检查
    适合验收，不适合替代以上三层

原则：能用确定性代码判断的，不要交给 LLM
同样要注意：Hook 可能递归触发（如 Stop Hook 启动子 Agent → 子 Agent 结束 → 再次触发 Stop Hook），需要 depth 限制和 once-per-run 控制。
