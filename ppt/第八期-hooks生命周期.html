// 第 2 页  —  从“钩子”这个意思切入
Hook 直译是钩子
重点不是钓鱼
是“挂上去”
生活里的挂钩，本身不做事。它只是预留一个位置，让你需要的时候，把钥匙、牌子、工具挂上去。
生活里墙上有挂钩，钥匙来了就挂上去。墙不用重建。
Agent 里流程里有挂点，日志、安全、审计来了就挂上去。主循环不用重写。

// 第 5 页  —  挂点放在哪里
触发位置
不是到处乱挂
只挂关键站点
Agent 一轮运行里，真正需要检查的地方不多。教学版保留五个常见位置，对应一轮对话从开始到结束的边界。
前
before_turn一轮开始前：注入上下文、限流、补 system prompt
后
after_turn模型返回后：记录耗时、保存模型输出
工

回
after_tool_call工具执行后：审计统计、整理结果、截断长输出
停
on_stop准备结束时：最后检查，必要时提醒再补一轮

// 第 8 页  —  安全边界
安全边界
安检必须
放在进闸前
很多人以为“有 Hook 就安全”。不对。Hook 只是挂点，安全不安全，看你把规则挂在哪里。
Before Hook

工具执行前检查。发现危险，直接不让过闸。

After Hook

工具执行后记录。事情已经发生，只能留证据。

稳
能用确定性规则判断的，不要交给模型猜。路径、金额、危险命令，优先写死规则。
三个误区
别把 Hook 神化
!
After Hook 不能阻止已经发生的事
审计不是安检，它只是记录。
!
异步 Hook 不适合授权
Agent 不等结果，适合日志，不适合付款审批。
!
LLM 不能当硬边界
模型判断有概率性，硬规则要放前面。
!
Stop Hook 要防递归
最多补一次，不要自己叫自己。
// 第 9 页  —  收尾复盘
最后记住三件东西
Hook 系统
三角色
Hook 不是单独工作。它要挂到 Registry 里，再由 Runner 在流程边界自动触发。
01
Hook
一张规则卡，写清楚某个挂点要做什么。
before_tool_call(ctx)
↓ register()
02
Registry
挂卡架，按注册顺序筛选和触发规则。
emit(event, ctx)
↓ runtime boundary
03
Runner
Agent 主流程，经过边界时把事件交给 Registry。
model -> tool -> result
一张 Hook 卡内部
四步跑完
一次检查
口播里的“三问一结果”，放到代码里就是四层模型。
ToolPolicyHook
matcher: write_file | run_command
Event什么时候触发？例如工具执行前：before_tool_call。
Matcher谁路过归我管？例如 write_file 和 run_command。
Handler具体怎么处理？检查路径、改写参数、拦危险命令。
Decision返回什么决定？allow、deny、ask、block。
一句话收尾：Hook 是规则卡，Registry 是挂卡架，Runner 是自动触发它们的主流程。
